Page Architecture Proposal

德州扑克训练器页面架构提案

目标不是先做漂亮页面，而是把对局强度、历史手牌沉淀和复盘准确性放在架构中心。页面只消费规则引擎、策略层、手牌库和复盘层输出的数据，避免 UI 自己推断规则或复盘结论。

核心原则

对局强度优先 AI 行为必须来自可替换策略层，不能散落在按钮和 UI 事件里。
复盘准确性优先 任意历史手牌都可复盘；复盘只使用 action-time snapshot，不用最终牌面、最终底池倒推。
页面服从数据 牌桌、手牌库、复盘、统计都读同一份 hand history 和 decision spots。

总体功能分区

牌桌训练纯对局，无策略提示
历史对局 / 手牌库会话、筛选、收藏、标记
复盘工作台选定手牌 / 选定决策点
数据中心长期 leak report
牌桌配置桌型、人数、盲注、买入
全局状态栏 当前训练模式、桌型、盲注、手数、保存状态
主工作区

牌桌时显示牌桌；手牌库显示历史列表和筛选；复盘时显示选中手牌的行动时间线和节点详情；数据中心显示图表和问题诊断。

上下文侧栏

牌桌时是行动按钮；手牌库是筛选器；复盘时是当前 spot 信息；数据页是维度选择。

底部抽屉 当前会话记录、手牌标签、导出 JSON、调试信息。

对局、收录、检索、复盘的数据流

1. 训练配置

User Intent 选择 HU / 6-max / 9-max、盲注、买入、AI 档位、训练目标。

2. Hand Engine

engine.js 发牌、盲注、合法动作、街次推进、摊牌结算。

3. Strategy Layer

后续实现 AI 只读取合法动作和当前状态，返回一个动作，不直接改状态。

4. Session Storage

hand archive 每手牌结束自动入库，按会话、桌型、标签和结果保存。

5. Review Workbench

review.js 用户从手牌库选择任意手牌，再抽取决策点生成复盘和 solver spot。

页面 1：牌桌训练

  • 只显示对局必要信息：底池、公共牌、玩家筹码、当前位置、轮到谁行动。
  • 不显示 GTO 建议、不显示复盘评价，避免训练时被提示污染。
  • 行动按钮来自 `getLegalActions()`，UI 不自己判断能不能加注。
  • AI 行动来自策略层，策略层只能提交合法 action。
  • 每手牌结束自动写入手牌库，供历史复盘、统计和导出使用。

页面 2：历史对局 / 手牌库

  • 按训练会话收录每一手牌，保留完整 hand history。
  • 支持按日期、桌型、人数、盲注、位置、手牌、输赢、标签筛选。
  • 支持收藏、标记、批量导出、重新打开某一手牌。
  • 从任意历史手牌进入复盘工作台，而不只复盘刚结束的一手。
  • 列表优先显示：手牌编号、时间、桌型、位置、起手牌、公共牌、净输赢、关键标签。

页面 3：复盘工作台

  • 复盘入口来自手牌库、当前手结束弹窗、数据中心某个 leak drill-down。
  • 按行动时间线展示 hero 的每个决策点，可跳转到任意街次。
  • 每个节点展示行动当时的底池、需跟、有效后手、公共牌、手牌。
  • 先展示“规则准确数据”，再展示策略评价。
  • GTO 接入后，每个节点可显示频率、EV、推荐动作和偏差。
  • 允许给手牌打标签：薄价值、过度跟注、错过 CBet、位置问题。

页面 4：数据中心

VPIP / PFR翻前松紧和主动性
3Bet / Fold to 3Bet翻前对抗质量
CBet / Fold to CBet翻后主动与防守
WTSD / W$SD / WWSF摊牌与赢池表现

数据中心不直接下结论，先给样本量和置信度。样本太少时只提示趋势，不做强判断。

页面 5：牌桌配置

单挑桌 HU 2 人桌，按钮位发小盲，适合训练高频对抗、宽范围和翻后单挑。
6-max 常规桌 6 人桌，默认主训练模式，接近线上现金桌常见结构。
9-max Full Ring 9 人桌，范围更紧，适合训练前位纪律和多人底池。
短码训练 20bb / 40bb，有更多 all-in 和 SPR 压缩场景。
标准现金桌 100bb 买入，盲注可选 10/20、20/40、50/100、100/200。
深筹码训练 150bb / 200bb，强调转牌、河牌大底池决策。
  • 可调参数：人数、按钮位置、大小盲、初始买入、有效后手、AI 席位数量、AI 档位。
  • 规则参数：No-Limit Hold'em 为主；ante、straddle、rake 暂时作为后续高级选项。
  • 训练主题：翻前入池、3Bet Pot、CBet、防守 CBet、河牌决策、短码 all-in。
  • 对局目标：固定手数、固定时间、直到指定 spot 出现、只练某类位置。

模块职责图

UI Views 牌桌、手牌库、复盘工作台、数据中心、牌桌配置。只负责展示和用户输入。
Session Controller 管理训练会话、当前手牌、桌型配置、页面跳转和保存策略。
Strategy Layer AI 策略接口。读取 engine state + legal actions，输出动作。GTO AI 最后讨论。
Hand Engine 唯一对局状态源。发牌、动作、街次、摊牌、结算、结构化 history。
Review Layer 从任意历史 hand history 抽取 decision spots，生成 review summary 和 solver spot。
Storage Layer 保存训练会话、hand history、统计指标、标签、训练配置和后续 solver 输出。

历史对局复盘流程

1. 自动收录

每手牌完成后保存 hand history、玩家配置、桌型、标签和结果。

2. 检索筛选

从手牌库按时间、桌型、位置、起手牌、输赢、行动类型筛选。

3. 选择复盘

打开某一手牌，选择整手复盘或跳到某个 hero 决策点。

4. 追踪问题

给手牌打标签，回写到数据中心，形成可重复训练的 leak 列表。

复盘节点数据结构

字段 用途 准确性要求
street / board 判断当时处于哪一街、公共牌是什么。 必须来自行动时快照，不能用最终 board。
potBefore / toCall 计算底池赔率、风险回报。 必须是行动前数值。
stackBefore / playerBetBefore 计算 SPR、有效后手、加注尺度。 必须是行动前数值。
action / targetBet / paid 记录真实选择，供复盘和统计。 必须和 hand history 一致。
solverOutput 后续接入频率、EV、推荐动作。 异步填充，不能影响原始 hand history。

开发顺序建议

A. UI 接入 engine

先让牌桌只使用 `engine.js` 状态，不再维护第二套状态机。

B. 牌桌配置模型

实现 HU / 6-max / 9-max、盲注、买入、AI 席位等配置。

C. 手牌库 / Storage

每手牌自动入库，支持历史检索、标签和导出。

D. 复盘工作台

从选定历史手牌提取 decision spots，先做基础准确复盘。

E. 策略层 / GTO

先拆 AI 策略层，最后接 solver 或策略表，填充频率和 EV。
